> **版本**: v1.0  
> **日期**: 2026-06-24  
> **适用范围**: Nice Today App "传统文化解读 + 健康生活助手" 双模块 AI Agent 集成  
> **数据截止**: 2026年6月24日（GitHub 实时数据）

---

## 一、执行摘要

**核心推荐方案（3句话）**：

1. **前端采用 Vercel AI SDK（v5.x）** 作为 Agent 交互层 —— 它是 React/TS 生态中成熟度最高、包体积最小（核心 ~150KB）、流式响应性能最优的方案，且原生支持 Tool Calling 的 TypeScript 类型推断，与 Capacitor WebView 的兼容性经社区广泛验证。

2. **后端采用 PydanticAI（主框架）+ LangGraph（复杂工作流）** 的双引擎架构 —— PydanticAI 以类型安全和依赖注入为核心设计，天然适配已有的 Python 计算模块（ephem 星盘、健康算法），且原生支持 MCP/A2A 协议；LangGraph 用于编排"计算→检索→生成"的多步状态机流程。

3. **记忆层采用 Mem0（通用记忆）+ 现有 sqlite-vec（向量检索）** 的混合方案 —— Mem0 是 2026 年 Agent 记忆领域最成熟的开源方案（57K Stars、YC S24、$42M B轮），其新增 TS SDK 可直接对接前端，Python SDK 对接后端，且支持自定义向量后端。

## 三、核心推荐详细卡片

### 🥇 推荐 #1：Vercel AI SDK（前端 Agent 交互层）

**项目信息**：
- GitHub: `vercel/ai` | Stars: ~30K+ | 许可: Apache 2.0
- 最新版本: v5.x（2026年6月）
- 核心维护方: Vercel 团队（Next.js 核心团队）

**为什么选它**：

1. **React/TS 生态最成熟** —— 提供 `useChat`、`useCompletion`、`useObject` 等开箱即用的 Hooks，流式响应实现业界最优，支持 ToolLoopAgent 的完整工具调用链路。
2. **Provider 无关** —— 通过统一的 `@ai-sdk/*` 适配器层，可无缝切换 OpenAI、Anthropic、Ollama（本地）等 Provider，天然支持"云端→离线"降级。
3. **Capacitor 兼容性** —— 核心基于标准 HTTP/WS 协议，无 DOM 依赖，在 WebView 中运行无阻。社区已有大量 Capacitor + AI SDK 的生产实践。
4. **包体积最优** —— 核心 `ai` 包仅 ~150KB，`@ai-sdk/react` ~50KB，远低于 Mastra（~300KB+）和 CopilotKit（~250KB）。

**与 Capacitor 的兼容性深度分析**：

| 维度 | 评估 | 说明 |
|------|------|------|
| WebView 环境 | ✅ 完全兼容 | 基于 fetch/WebSocket，无 Node.js 依赖 |
| CORS 处理 | ✅ 通过 Capacitor HTTP 插件 | 可使用 `@capacitor/http` 绕过 CORS |
| 原生桥接 | ✅ 通过 Tool Calling | 工具调用可桥接 Capacitor 原生插件 |
| 内存占用 | ✅ 低 | 长会话使用增量 Streaming，无全量缓存 |
| 首屏加载 | ✅ 快 | Tree-shaking 友好，按需引入 |

**流式渲染性能**（移动端 WebView）：
- 基于 ReadableStream API，无需 polyfill
- Token-by-token 渲染，TTFB（首 Token 延迟）< 200ms
- 支持 abort/取消，适配移动端"滑动离开即停止"交互

**工具调用类型安全示例**：
```typescript
import { tool } from 'ai';
import { z } from 'zod';

// TypeScript 自动推断参数类型
const calculateChart = tool({
  description: '计算紫微斗数命盘',
  parameters: z.object({
    birthDate: z.string().describe('出生日期 YYYY-MM-DD'),
    birthTime: z.string().describe('出生时辰'),
    gender: z.enum(['male', 'female']),
  }),
  execute: async ({ birthDate, birthTime, gender }) => {
    // 桥接 Capacitor 原生层调用 Python 后端
    const result = await fetch('/api/chart/calculate', {
      method: 'POST',
      body: JSON.stringify({ birthDate, birthTime, gender }),
    });
    return result.json();
  },
});
```

### 🥇 推荐 #2：PydanticAI（Python 后端主框架）

**项目信息**：
- GitHub: `pydantic/pydantic-ai` | Stars: ~15K+ | 许可: MIT
- 维护方: Pydantic 团队（Samuel Colvin）
- 最新版本: v0.4+（2026年6月，GA 版本）

**为什么选它**：

1. **类型安全是核心设计** —— 不是事后添加，而是从第一行代码就开始。所有 Agent 输入/输出、工具参数、依赖注入都有完整类型推断。
2. **依赖注入（Dependency Injection）** —— 这是 PydanticAI 独有的杀手级特性。可以将已有的 Python 计算模块（ephem 星盘、健康算法）直接注入 Agent，无需重构：
   ```python
   from dataclasses import dataclass
   from pydantic_ai import Agent
   
   @dataclass
   class ChartDeps:
       ephemeris: EphemerisCalculator  # 已有模块
       vector_store: SQLiteVecStore    # 已有模块
   
   chart_agent = Agent(
       'openai:gpt-4o',
       deps_type=ChartDeps,
       system_prompt='你是紫微斗数解读专家...'
   )
   ```
3. **原生 MCP + A2A 支持** —— 可将现有 Python 计算模块封装为 MCP Server，供其他 Agent 调用。
4. **Graph 支持** —— 内置有向图（DAG）编排，支持复杂多步工作流，替代 LangGraph 的大部分场景。
5. **Pydantic Logfire** —— 配套的可观测平台，免费自托管，Tracing + Evaluation 一体化。

**与现有 Python 模块的集成方式**：

```python
# 现有模块（无需修改）
class HealthAlgorithm:
    def analyze_sleep(self, data: dict) -> dict: ...
    def generate_advice(self, profile: dict) -> str: ...

# PydanticAI 集成（通过依赖注入）
@dataclass
class HealthDeps:
    health_algo: HealthAlgorithm      # 直接注入
    solar_terms: SolarTermService     # 节气数据服务
    user_store: UserHealthStore       # 用户健康数据

health_agent = Agent(
    'openai:gpt-4o',
    deps_type=HealthDeps,
    result_type=HealthAdvice,  # 结构化输出，Pydantic 模型约束
)

@health_agent.tool
async def get_sleep_analysis(ctx: RunContext[HealthDeps], days: int = 7):
    data = await ctx.deps.user_store.get_recent_sleep(days)
    return ctx.deps.health_algo.analyze_sleep(data)
```

**记忆持久化方案**：
- 通过依赖注入可接入任意存储后端
- 推荐方案：Mem0 Python SDK → 统一记忆层
- 降级方案：直接操作 sqlite-vec 做向量检索

**多 Agent 通信开销**：
- 同进程内：函数调用级别，延迟 < 1ms
- 跨服务：HTTP/gRPC，延迟 10-50ms
- Agent Graph 内部：状态机调度，无额外网络开销

### 🥇 推荐 #3：Mem0（通用记忆层）

**项目信息**：
- GitHub: `mem0ai/mem0` | Stars: ~57K+ | 许可: Apache 2.0
- 融资: YC S24，$42M B轮（2026年）
- 最新版本: v3（2026年4月，新记忆算法）

**为什么选它**：

1. **2026 年 Agent 记忆领域的事实标准** —— LoCoMo 基准 91.6 分，LongMemEval 94.8 分，均远超竞品。
2. **双端 SDK** —— Python (`mem0ai`) + TypeScript (`mem0ai`) SDK 同时可用，天然适配 Nice Today 的混合架构。
3. **自托管 + 云平台** —— 支持 `docker compose up` 一键自托管，也提供零运维云平台。
4. **多级记忆** —— User Memory（用户偏好）、Session Memory（会话上下文）、Agent Memory（Agent 状态）三级记忆，完美适配"健康+玄学"双场景。
5. **新算法特性**：
   - ADD-only 提取（单次 LLM 调用，无 UPDATE/DELETE）
   - 实体链接（跨记忆实体关联增强检索）
   - 时序推理（按时间排序检索正确版本的事实）
   - 多信号融合（语义 + BM25 + 实体匹配并行评分）

**与 sqlite-vec 的兼容性**：
- Mem0 默认使用 Qdrant 作为向量后端
- 可通过自定义 `VectorStore` 类接入 sqlite-vec
- 推荐方案：Mem0 的抽象记忆层 + sqlite-vec 作为向量存储后端

**会话记忆压缩**：
- Mem0 v3 的 ADD-only 算法天然压缩：只提取关键事实，不存储原始对话
- Token 效率：单次检索仅消耗 ~7K tokens（含完整用户画像）
- 长对话不爆炸：记忆与对话分离，对话历史可滚动清理

## 四、Nice Today 场景映射

### 场景 1：紫微斗数/奇门遁甲解读 Agent

**推荐技术组合**：

```
用户输入 → Vercel AI SDK (useChat, 流式)
         → PydanticAI 后端 Agent
         → Tool: ephemeris.calculate_chart()     [Python 计算模块]
         → Tool: vector_store.search_classics()   [sqlite-vec RAG]
         → Mem0: 加载用户历史命盘 & 追问上下文
         → 结构化输出 (Pydantic 模型约束)
         → 流式返回前端
```

**流水线编排评估**：

| 步骤 | 框架 | 是否阻塞 | 说明 |
|------|------|---------|------|
| 意图识别 | PydanticAI Agent | 否 | LLM 快速分类 |
| 命盘计算 | Tool → ephem | 是（CPU） | 异步执行，~100ms |
| 经典检索 | Tool → sqlite-vec | 否 | 向量检索，~50ms |
| 解读生成 | PydanticAI Agent | 是（LLM） | 流式输出，~3-8s |
| 记忆存储 | Mem0 | 否 | 后台异步 |

**命盘结果缓存策略**：
```python
# 使用 Mem0 的 User Memory 缓存命盘结果
async def calculate_and_cache_chart(user_id: str, birth_data: dict):
    # 1. 先查缓存
    cached = await memory.search(
        query="命盘计算结果",
        filters={"user_id": user_id, "metadata.type": "chart_result"},
    )
    if cached["results"]:
        return json.loads(cached["results"][0]["memory"])
    
    # 2. 缓存未命中，计算命盘
    chart = ephemeris.calculate(birth_data)
    
    # 3. 存入记忆
    await memory.add(
        [{"role": "assistant", "content": json.dumps(chart)}],
        user_id=user_id,
        metadata={"type": "chart_result", "calc_date": datetime.now().isoformat()},
    )
    return chart
```

**追问支持**：
- Mem0 自动维护对话上下文（Session Memory）
- 用户问"我的事业宫怎么看？"时，Agent 可从记忆中检索命盘数据
- 无需重复计算，直接引用缓存结果

```typescript
    const advice = JSON.parse(message.content);
    if (advice.action_items.some(item => item.reminder)) {
      // 桥接 Capacitor 本地通知
      LocalNotifications.schedule({
        notifications: advice.action_items
          .filter(item => item.reminder)
          .map(item => ({
            title: '健康提醒',
            body: item.text,
            schedule: { at: new Date(item.reminder_time) },
          }))
      });
    }
  }
});
```

**安全边界设计**：
- 系统提示词明确禁止医疗建议
- 输出 Schema 强制包含 `risk_disclaimer` 字段
- 敏感关键词（"药"、"症"、"病"）触发安全拦截器
- 用户反馈闭环：点赞/点踩 → 标记为训练信号

### 场景 3：多 Agent 调度中枢

**推荐技术组合**：

**冲突解决机制**：
```python
# 当健康 Agent 和玄学 Agent 建议冲突时
class ConflictResolver:
    def resolve(self, health_advice: str, astro_advice: str) -> str:
        """优先级：健康安全 > 玄学建议"""
        # 1. 检查是否存在安全冲突
        conflicts = self.detect_conflicts(health_advice, astro_advice)
        if conflicts:
            # 2. 健康建议优先，玄学建议降为"参考"
            return self.merge_with_priority(
                primary=health_advice,
                secondary=astro_advice,
                label_primary="健康建议",
                label_secondary="命理参考"
            )
        # 3. 无冲突，正常合并
        return self.merge_normal(health_advice, astro_advice)
```

- 路由不确定时（confidence < 0.7）→ 前端展示选择器让用户手动选择
- Agent 执行中 → 支持用户发送"切换到健康模式"等指令
- 结果不满意 → "重新生成"或"换个角度"按钮触发重试

```python
# backend/agents/orchestrator.py
from pydantic_ai import Agent, RunContext
from pydantic import BaseModel
from dataclasses import dataclass
from typing import Literal

# ---- 依赖定义 ----
@dataclass
class AgentDeps:
    mem0: 'Memory'                    # Mem0 客户端
    ephemeris: 'EphemerisCalculator'  # 星盘计算
    health_algo: 'HealthAlgorithm'    # 健康算法
    vector_store: 'SQLiteVecStore'    # 向量检索
    user_id: str

# ---- 意图分类 ----
class IntentResult(BaseModel):
    intent: Literal["health", "chart", "general", "mixed"]
    confidence: float

classifier = Agent(
    'openai:gpt-4o-mini',
    result_type=IntentResult,
    system_prompt='判断用户意图：health=健康养生, chart=命理解读, general=其他, mixed=混合'
)

# ---- 健康 Agent ----
class HealthAdvice(BaseModel):
    category: Literal["sleep", "exercise", "diet", "emotion"]
    suggestion: str
    action_items: list[str]
    solar_term_tip: str | None = None
    risk_disclaimer: str = "本建议仅供参考，不构成医疗建议"

health_agent = Agent(
    'openai:gpt-4o',
    deps_type=AgentDeps,
    result_type=HealthAdvice,
    system_prompt='你是 Nice Today 健康生活顾问，结合用户数据和节气提供个性化建议。'
)

@health_agent.tool
async def get_user_health_data(ctx: RunContext[AgentDeps], days: int = 7):
    """获取用户近期健康数据"""
    memories = await ctx.deps.mem0.search(
        query="健康数据 睡眠 运动",
        filters={"user_id": ctx.deps.user_id},
        top_k=10,
    )
    return [m["memory"] for m in memories["results"]]

@health_agent.tool
async def get_solar_term_advice(ctx: RunContext[AgentDeps]):
    """获取当前节气的养生建议"""
    from datetime import date
    current_term = get_current_solar_term(date.today())
    return ctx.deps.vector_store.search(
        query=f"{current_term} 养生 建议",
        collection="solar_terms",
        top_k=3,
    )

# ---- 命理 Agent ----
class ChartReading(BaseModel):
    summary: str
    palace_details: dict[str, str]
    advice: str
    classics_references: list[str]

chart_agent = Agent(
    'openai:gpt-4o',
    deps_type=AgentDeps,
    result_type=ChartReading,
    system_prompt='你是紫微斗数解读大师，精通经典古籍，用现代语言解读命盘。'
)

@chart_agent.tool
async def calculate_chart(ctx: RunContext[AgentDeps], birth_date: str, birth_time: str, gender: str):
    """计算紫微斗数命盘"""
    return ctx.deps.ephemeris.calculate(birth_date, birth_time, gender)

@chart_agent.tool
async def search_classics(ctx: RunContext[AgentDeps], query: str):
    """检索经典古籍中的相关记载"""
    return ctx.deps.vector_store.search(
        query=query,
        collection="classics_md",
        top_k=5,
    )

# ---- 调度中枢 ----
class OrchestratorResponse(BaseModel):
    agent_used: str
    response: str
    follow_up_suggestions: list[str]

async def orchestrate(user_message: str, deps: AgentDeps) -> OrchestratorResponse:
    """主调度函数"""
    # 1. 加载用户记忆
    user_memories = await deps.mem0.search(
        query=user_message,
        filters={"user_id": deps.user_id},
        top_k=5,
    )
    memory_context = "\n".join(m["memory"] for m in user_memories["results"])
    
    # 2. 意图分类
    intent_result = await classifier.run(
        f"用户记忆上下文：{memory_context}\n\n用户消息：{user_message}"
    )
    intent = intent_result.data
    
    # 3. 路由到对应 Agent
    if intent.confidence < 0.7:
        # 低置信度，返回让用户选择
        return OrchestratorResponse(
            agent_used="classifier",
            response="我不太确定你想问什么，请选择：\n1. 🏥 健康咨询\n2. 🔮 命理解读",
            follow_up_suggestions=["健康养生", "紫微斗数", "综合运势"],
        )
    
    if intent.intent == "health":
        result = await health_agent.run(user_message, deps=deps)
        agent_used = "health"
    elif intent.intent == "chart":
        result = await chart_agent.run(user_message, deps=deps)
        agent_used = "chart"
    else:
        # 通用对话，直接用 LLM
        result = await health_agent.run(user_message, deps=deps)  # fallback
        agent_used = "general"
    
    # 4. 存储对话记忆（异步，不阻塞响应）
    await deps.mem0.add(
        [
            {"role": "user", "content": user_message},
            {"role": "assistant", "content": result.data.model_dump_json()},
        ],
        user_id=deps.user_id,
    )
    
    return OrchestratorResponse(
        agent_used=agent_used,
        response=result.data.model_dump_json(),
        follow_up_suggestions=[],
    )
```

| **CrewAI "自主协作"** | ⚠️ 生产稳定性 | "自主 Agent 协作"在生产中表现不稳定，复杂场景调试困难 | **生产环境慎用 CrewAI 的全自动模式** |
| **Web-LLM 移动端** | ⚠️ WebGPU 兼容 | Android WebView 对 WebGPU 支持不完整，iOS Safari 限制严格 | **不作为移动端方案** |
| **MediaPipe LLM 中文** | ⚠️ 中文能力差 | 主要优化英文，中文传统文化术语理解差 | **不推荐** |

### ⚠️ 中风险关注

| 项目 | 风险 | 说明 | 建议 |
|------|------|------|------|
| **Mastra** | 包体积大 | ~300KB+ 核心，移动端首屏加载需关注 | 按需引入，Tree-shaking |
| **CopilotKit** | 移动端 UI | Generative UI 组件在小屏上需定制 | 如选 CopilotKit，需额外 UI 适配工作 |
| **Mem0 自托管** | 运维复杂度 | 需要 PostgreSQL + Qdrant + Redis | 初期用 Cloud，后期迁移自托管 |
| **Graphiti** | 依赖重 | 需要 Neo4j 或 FalkorDB（图数据库） | 评估是否真正需要图结构记忆 |
| **ONNX Runtime Capacitor** | 社区插件质量 | 没有官方 Capacitor 插件，需自研或找社区 | **明确标注"需自研"** |
| **PydanticAI Graph** | 新功能 | Graph 编排是较新功能，社区实践少 | 关注 Issue 动态，准备好降级方案 |

### 📋 已知 Capacitor/WebView 痛点

| 痛点 | 影响 | 缓解方案 |
|------|------|---------|
| SSE 长连接不稳定 | 流式响应可能中断 | WebSocket 降级 + 自动重连 |
| WebGPU 不可用 | 端侧推理受限 | 使用 ONNX Runtime WASM 后端（慢但可用） |
| 内存限制 | 端侧大模型 OOM | 模型量化 + 增量加载 + 内存监控 |
| CORS 限制 | API 调用被拦截 | Capacitor HTTP 插件绕过 |
| 后台挂起 | 定时任务被系统杀掉 | Capacitor Background Task 插件 |

### 💡 需自研的组件

| 组件 | 说明 | 参考实现 |
|------|------|---------|
| Capacitor ONNX Runtime 插件 | 封装 ONNX Runtime Mobile 为 Capacitor 插件 | 参考 `@nice-app/capacitor-onnx` |
| 端侧 RAG Pipeline | sqlite-vec + 端侧 Embedding 的完整检索链 | 参考 LlamaIndex Python 的 RAG 逻辑 |
| MCP Server（节气/星历/穴位） | 封装现有 Python 计算模块为 MCP 工具 | 参考 MCP Python SDK 示例 |
| 健康建议安全过滤器 | 拦截医疗敏感词，添加免责声明 | 自定义 Guardrails |
