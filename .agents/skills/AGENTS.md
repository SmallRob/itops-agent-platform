# 全局AI行为规范

## 核心原则：思考-提问-验证

### 1. 写代码前必须思考
- **理解意图**：先分析用户需求的真实目的，而非表面描述
- **阅读上下文**：查看相关文件、函数调用链、现有实现
- **识别影响范围**：评估修改会影响哪些模块和功能

### 2. 必须提问，绝不瞎猜
遇到以下情况**必须主动提问**：
- 需求不明确或有多种理解方式
- 不确定使用哪种技术方案或设计模式
- 不清楚业务逻辑或边界条件
- 发现现有代码可能有其他依赖关系
- 需要修改但不确定用户期望的行为

**禁止行为**：
- ❌ 假设用户的意图
- ❌ 自行决定技术选型（除非项目已有明确规范）
- ❌ 忽略潜在的副作用
- ❌ 编造不存在的API或函数

### 3. 修改代码时：简洁优先
- **最小改动原则**：只修改必要部分，不做"顺手"重构
- **保持风格一致**：遵循项目现有的代码风格和命名规范
- **不添加多余代码**：不加注释（除非要求）、不加日志、不加防御性代码（除非必要）
- **精确编辑**：使用精确的字符串替换，而非重写整个文件

### 4. 验证流程
完成修改后：
1. 运行项目的lint/typecheck命令验证代码质量
2. 如果有测试，确认测试通过
3. 简要说明修改内容和原因

## 工作流程模板

```
1. 思考 → 理解需求，分析影响
2. 提问 → 澄清不确定点
3. 实施 → 最小化修改
4. 验证 → lint/typecheck/测试
5. 总结 → 简要说明改动
```

## 特殊场景处理

### 代码审查时
- 指出具体问题，给出修改建议
- 不做未经请求的重构

### 调试时
- 先复现问题，确认根因
- 提出修复方案，获得确认后再修改

### 新功能开发时
- 先讨论设计方案
- 确认接口和数据结构
- 分步实现，逐步验证

# UTF-8 与中文文件处理最高优先级规则

本节优先级极高，适用于所有涉及中文文件、脚本和文档的读写操作，并与上方事故复盘规则同级执行。

- 所有文件读写默认使用 UTF-8 编码，修改文件时不得改变原有编码、换行风格和无关内容。
- 在 PowerShell 中读取含中文文件前，先执行 `chcp 65001`，并设置：
  - `[Console]::OutputEncoding = [System。Text。Encoding]::UTF8`
  - `$OutputEncoding = [System。Text。Encoding]::UTF8`
- 读取含中文文件时优先使用 `Get-Content -Raw -Encoding UTF8`。
- 禁止用 PowerShell 的 here-string 管道、重定向、`Set-Content`、`Out-File` 写入含中文源码、JSON 或文档。
- 不要用 `sed`、`awk` 处理含中文文件，改用 Python 或 Node。js。
- 使用 Python 或 Node。js 脚本处理中文文件时，必须显式以 UTF-8 读写。
- 代码注释使用中文。
